# 📄 PYQ Website — Implementation Plan

## 1. Problem Statement

Your college hosts PYQ papers on an FTP server that:
- Doesn't work in normal browsers
- Requires File Explorer or a dedicated FTP client
- Has a confusing folder structure
- Previous attempt on Render suffered from extreme cold-start latency

**Goal:** Build a clean, fast website that indexes the FTP content and lets students browse/download papers effortlessly.

---

## 2. Why the Previous Version Was Slow

```mermaid
graph LR
    A[User Request] --> B[Render Wakes Up<br/>~30-60s cold start]
    B --> C[FastAPI connects to FTP<br/>~5-10s per request]
    C --> D[FTP lists directory<br/>~2-5s]
    D --> E[Response to user]
    style B fill:#ff6b6b,color:#fff
    style C fill:#ff6b6b,color:#fff
```

Two compounding problems:
1. **Render free tier cold starts** — server spins down after 15 min of inactivity, takes 30-60s to wake up
2. **Live FTP fetching** — every page load hit the FTP server in real-time (slow, unreliable)

---

## 3. The Fix: Offline Sync + Pre-built Index

Instead of hitting FTP on every user request, we **sync once** and serve cached data:

```mermaid
graph TB
    subgraph "One-time / Periodic Sync"
        S1[sync_ftp.py] --> S2[Connect to FTP]
        S2 --> S3[Walk all directories]
        S3 --> S4[Download all PDFs<br/>to local /papers folder]
        S4 --> S5[Build index in<br/>SQLite database]
    end

    subgraph "User Request Flow — Fast!"
        U1[User visits site] --> U2[FastAPI reads<br/>SQLite index]
        U2 --> U3[Returns file list<br/>instantly ~50ms]
        U3 --> U4[User clicks download]
        U4 --> U5[Serve PDF from<br/>local disk]
    end

    style S1 fill:#4ecdc4,color:#fff
    style U3 fill:#45b7d1,color:#fff
```

> [!IMPORTANT]
> This is the **key architectural decision** that makes this project interview-worthy. You can explain:
> - "I identified that the bottleneck was real-time FTP access on every request"
> - "I designed an offline sync pipeline that decouples data ingestion from serving"
> - "This reduced response times from 30-60s to under 100ms"

## 4. Hosting Alternatives

| Option | Cold Start | Cost | Persistent Storage | Recommendation |
|--------|-----------|------|-------------------|----------------|
| **Render (Free)** | 30-60s ❌ | Free | No ❌ | Previous choice — avoid |
| **Railway (Free)** | ~5s | Free (500 hrs/mo) | Yes ✅ | Good option |
| **Oracle Cloud Free VM** | None ✅ | Always free | Yes ✅ | ⭐ Best for this project |
| **Fly.io (Free)** | ~3s | Free (3 shared VMs) | Volumes ✅ | Good option |
| **VPS (DigitalOcean/Hetzner)** | None ✅ | ~$4-5/mo | Yes ✅ | Most reliable |
| **GitHub Pages + API** | None ✅ | Free | Via Git ✅ | Hybrid approach |

### Recommended: **Two-tier approach**

1. **Sync & store PDFs in the Git repo itself** (if total size < 500MB) or in **Cloudflare R2** (free 10GB)
2. **Host the FastAPI app on Railway or Oracle Cloud** — no cold starts, persistent disk

> [!TIP]
> For a college project, if total PDF size is manageable (< 200-300 MB), storing them directly in the repo and deploying on Railway is the simplest and most reliable approach.

## 8. Database Schema

```sql
CREATE TABLE papers (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    filename    TEXT NOT NULL,           -- "CS301_2024_EndSem.pdf"
    filepath    TEXT NOT NULL,           -- relative path in /data/papers/
    ftp_path    TEXT NOT NULL UNIQUE,    -- original FTP path (for dedup)
    department  TEXT,                    -- "CSE", "ECE", etc.
    subject     TEXT,                    -- "Data Structures"
    year        TEXT,                    -- "2024"
    exam_type   TEXT,                    -- "EndSem", "MidSem", "Quiz"
    file_size   INTEGER,                -- bytes
    synced_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_department ON papers(department);
CREATE INDEX idx_subject ON papers(subject);
CREATE INDEX idx_year ON papers(year);
```

> [!IMPORTANT]
> The exact schema for `department`, `subject`, `year`, and `exam_type` depends on how your FTP folder structure is organized. **I need you to describe the FTP folder hierarchy** (e.g., is it organized like `Department/Subject/Year/file.pdf`?).
